# Results — browser tool comparison

Scenario: `store-scenario.md`. Graded against `store-scenario-answer-key.md`. See `plan.md` for methodology.

## Methodology note (read before the table)

**The 9 runs were not actually isolated from each other**, despite each being a fresh subagent session. All 9 were launched in a single parallel batch, so multiple agents on the same tool shared the same underlying browser/MCP server process, and cross-run interference showed up constantly:

- Playwright run 3: cart pre-populated with items before it acted, sort order and cart badge changed with no action taken, spontaneous logout mid-flow.
- Playwright run 2: cart pre-populated before it acted (leftover state).
- Chrome DevTools run 2: detected cart contents changing and unexplained navigation, correctly diagnosed it as a shared/contended browser context, and proactively opened an isolated context (`isolatedContext` param) to redo the run cleanly.
- Chrome DevTools run 3: discovered a second tab mid-run it didn't create — another concurrent run sharing the same browser — causing several actions to silently land on the wrong tab.
- Claude in Chrome run 2: saw inconsistent product images across logins of the same test user and correctly declined to report it as a confirmed finding, suspecting session bleed-through.

This inflates tool-call counts, tokens, and wall-clock time for every tool — it's an artifact of running the comparison in parallel, not a fair reflection of intrinsic tool efficiency. **Chrome DevTools MCP was the only tool where a run self-diagnosed the contamination and fixed it via an explicit isolated-context feature** — that's a real capability signal, not noise, and is called out in the verdict below. Treat the raw efficiency numbers as upper bounds, not clean baselines; a rerun with runs staggered sequentially (or verified separate browser processes per run) would give cleaner numbers.

Separately: a native "password found in a data breach" popup appeared during testing (since `secret_sauce` is a public, well-known test password) — **and per user observation, this happened across Playwright and Chrome DevTools MCP runs too, not just Claude in Chrome.** That rules out the profile-isolation explanation: it's most likely Chrome's real-time password-leak check, which hashes any *submitted* password against a known-breach list independent of whether that password is saved in the profile or which tool is driving the browser. Correction to an earlier draft of this note: this is scenario-level noise (the deliberately "leaked" test password), not a Claude-in-Chrome-specific cost, and a fresh/clean Chrome profile for Claude in Chrome would likely not have prevented it. No run reported the popup blocking an action either way.

What *does* still look Claude-in-Chrome-specific, since no Playwright/Chrome DevTools transcript reported it: Chrome's autofill repeatedly duplicating/overwriting typed text, and once feeding a stale username into the login form (a real task failure, worked around with `form_input`). That's plausibly tied to running against a real, already-populated browser profile rather than a clean instance.

Also: the Agent tool's completion notification only reports one combined `subagent_tokens` figure, not separate input/output token counts as the plan specified — the table below merges those two columns into one "Tokens" column.

## Verdict

**What the data actually supports:**

- **Correctness ties across all three tools.** All 9 runs scored 8/8 — none missed the image bug, none missed the totals, none glossed over the performance delay. This scenario doesn't differentiate tools on correctness.
- **Claude in Chrome is the costliest tool, for a real reason, not a contamination artifact.** It used roughly 2x the tokens of the other two on average (~132k vs ~69-78k) and the most tool calls (~124 avg vs ~69-72). Its overhead traces to running in a real, logged-in Chrome profile: autofill repeatedly duplicated/overwrote typed text and once fed a stale username into a login form, actively causing a task failure that had to be worked around with `form_input`. That's an intrinsic property of automating a real browser profile, not something a clean re-run would erase.
- **Chrome DevTools MCP vs Playwright MCP is genuinely undecided from this run — not "roughly tied," actually unmeasured.** Their tool-call/token gap (68.7 vs 71.7 calls, 77.5k vs 69.0k tokens) falls well inside the noise created by running all 9 agents in parallel and sharing browser state (see methodology note above). Worse, the one apparent differentiator — Chrome DevTools run 2 self-diagnosing contention and recovering via `isolatedContext` — is circular evidence: that feature only got exercised *because* the parallel-execution harness created the contamination in the first place. A sequential run has no contention to detect, so that "reliability edge" would likely never surface at all. It should not be used to justify a default.

**No default recommendation between Chrome DevTools MCP and Playwright MCP yet.** That comparison needs a clean sequential re-run (one tool fully finishing before the next starts, no shared browser contention) before it can support a "reach for this by default" claim — tracked as an open item in `plan.md`.

**Where Claude in Chrome earns its cost**: tasks that require a real authenticated browser profile/session (things a clean headless context can't do), or bugs that are only detectable visually and not via the DOM. This scenario's image bug happened to be DOM-detectable via identical `img.src`, so Claude in Chrome's vision advantage was never actually exercised here — a genuinely pixel-only bug (e.g. a CSS layout overlap with no DOM signal) would be a fairer test of that strength.

## Sequential re-run (2026-07-08) — improved but still confounded

Ran all 9 runs strictly sequentially this time (one subagent fully finishes before the next starts, tool-by-tool: Playwright ×3, then Chrome DevTools ×3, then Claude in Chrome ×3), which eliminates the *parallel*-contention problem from the 2026-07-03 batch (no more stray tabs, no more state changing mid-action from another run). `.mcp.json` also had `--isolated` added to the Playwright and Chrome DevTools MCP server args, and each tester agent's instructions were updated to check for and clear leftover state (Reset App State + Logout) before step 1.

**This is not a clean baseline — it surfaced a narrower version of the same problem.** The Playwright and Chrome DevTools MCP servers are launched once per Claude Code session (per `.mcp.json`), not once per Agent call, so the underlying browser process persists across sequential runs of the *same* tool even with `--isolated`. Playwright runs 2 and 3 both found the cart pre-populated with 2 items immediately after a fresh login (leftover from the prior run), despite landing on the login form (not an authenticated session) — the login/logout boundary reset auth state but not cart localStorage. Each run detected this via the mandated pre-flight check and self-corrected with an explicit Reset App State + reload before proceeding. **Practical effect: only run 1 of each tool (Playwright, Chrome DevTools) is a genuinely clean start — runs 2 and 3 carry extra reset/recovery tool calls baked into their totals.** Claude in Chrome has no isolation mechanism at all (it drives a real persistent Chrome profile by design), so this class of leftover-state issue applies to it on every run, same as before.

For interactive use where a human is present and just verifying an implemented feature, this isn't worth the setup overhead — a person notices and dismisses an overlay in a second, and the browser has other extensions installed for a reason.

**The flakiness recurred in all 3 follow-up runs — it's a real, repeatable property of Chrome DevTools MCP's native `click`, not a one-off.** Every run hit at least one "tool reports success, but the DOM didn't change" click, confirmed independently via `evaluate_script`/snapshot each time — never a `fill`/`fill_form` issue, always `click`. What varies is *severity*: run 1 needed a DOM-manipulation workaround for nearly every interaction on the page (add-to-cart, checkout fields, checkout/continue buttons, the burger menu), while runs 4-6 needed only 1-2 retries on a single click before succeeding normally. So "Chrome DevTools MCP's click is unreliable" is now a confirmed, generalizable finding — but "run 1's specific severity" does not generalize; it looks like the tail end of a variance distribution, not the typical case.

**A second, separate pattern is not flakiness — it's expected.** In every run across both batches (original 3 + these 3), the login-button click for `performance_glitch_user` throws an interactivity-timeout error, yet the login always succeeds once checked. This is the tool's click-stability-wait being exceeded by the test user's deliberately injected delay, not a broken click — don't lump it in with the real no-op flakiness above when judging reliability.

### Verdict

**Correctness still ties, 8/8 across all 9 runs.** Same conclusion as the contaminated batch: this scenario doesn't differentiate the three tools on correctness. All 9 runs matched the answer key exactly on steps 2, 4, 6, and named the image-swap bug on step 7; step 8 passed the qualitative bar (flag the slowdown) on all 9, though the actual number reported ranged from a clean ~5.0s to an inflated ~16s depending on measurement method.

**Playwright vs. Chrome DevTools MCP is still undecided — and for a different reason than last time.** Last batch it was undecided because of parallel-run contamination; this batch it's undecided because (a) the metrics disagree with each other, and (b) the leftover-state-bleed issue means only run 1 of each tool is a clean sample size of one. Tool calls favor Playwright (42.7 vs 62.0, or 53.5 excluding DevTools' run-1 outlier). Tokens favor Chrome DevTools once that outlier is excluded (~78k vs ~93k) but favor Playwright if it's included. That's not "roughly tied," it's genuinely unresolved — the two metrics point in opposite directions depending on which sample you trust. **No default recommendation between these two tools.**

**Chrome DevTools MCP's native-click flakiness is confirmed, but run 1's severity isn't.** A same-day 3-run follow-up (below) reproduced the "tool reports success, DOM didn't change" failure mode in every run — it's real and repeatable. Run 1's outlier magnitude (needing a workaround on nearly every interaction) did not recur; runs 2-3-and-the-follow-up needed only 1-2 retries per run. Treat "click is somewhat flaky" as confirmed and "run 1's severity is typical" as false.

**Claude in Chrome's cost is now measured on a clean, extension-free profile** — see the note under the results table above for why, and the recommendation on when a clean profile actually matters. With that confound removed, Claude in Chrome's numbers (46.7 calls / 78,695 tokens / ~9m48s) land roughly in the same range as Chrome DevTools MCP's sequential-batch numbers (62.0 calls / 131,421 tokens with the run-1 outlier, or 53.5 calls / ~78,000 tokens excluding it) and closer to Playwright's (42.7 calls / 92,786 tokens) — the three-way cost gap seen in the original parallel batch was driven substantially by contamination and profile artifacts, not a fundamental efficiency difference between the tools.

A different tool quirk showed up in these clean-profile runs: **raw JS form-filling fights the app's controlled inputs.** Runs 2 and 3 both hit the same failure mode — setting an `<input>`'s `.value` via `javascript_tool` and dispatching a synthetic `input` event doesn't register with saucedemo's (React-style) controlled-input state, so the app treats the field as still empty. Both recovered by falling back to normal UI-driven click+type. This didn't happen in run 1, which used `find` + `form_input` instead of raw JS — suggests `form_input`/UI-driven typing is the more reliable path for this tool on this kind of app, and raw `element.value` assignment should be a last resort, not a default.

**What would actually settle Playwright vs. Chrome DevTools MCP:** true per-run isolation, which needs either the MCP server process itself restarted between runs (not just `--isolated` server args, since the server/browser persists for the life of the Claude Code session) or an explicit teardown step in each tester's instructions that clears cookies/localStorage/all open pages, not just clicking "Reset App State" (which only resets in-app cart state, not the browser-level state a truly fresh session would have). Still no clean three-way head-to-head — Playwright and Chrome DevTools MCP have not been re-run against a similarly isolated setup. Not yet scheduled.
